# Idea Refine

## Overview

结构化发散-收敛思维：将模糊的概念转化为具体的提案。不同于 interview-me（通过提问从用户那里提取需求），idea-refine 是你自己或与用户一起对一个想法进行多角度探索、筛选和收敛，最终得到一个可执行的方案。适用于创意阶段、方案选型、架构设计等需要探索多种可能性的场景。

## When to Use

- 有一个粗略概念但不确定具体怎么做
- 需要探索多种方案再选择最佳路径
- 架构设计、技术选型、产品创意阶段
- 用户说"我有个想法，你帮我琢磨琢磨"
- NOT for 需求已经明确的任务（用 spec-driven-development）
- NOT for 需求模糊但用户有答案（用 interview-me）

## Core Process

### Phase 1: Divergent（发散）

目标是生成尽可能多的可能性，不做筛选和评判。

1. **设定探索框架** — 明确探索的边界和方向
   - "我们要解决什么问题？"
   - "有哪些约束不可突破？"

2. **自由生成** — 列出所有可能的方案/路径
   - 不考虑可行性，不考虑资源限制
   - 至少列出 5 种不同方向
   - 使用思维导图或列表形式

3. **扩展边界** — 有意挑战假设
   - "如果预算翻倍会怎样？"
   - "如果时间只有一半呢？"
   - "最疯狂的做法是什么？"

**发散阶段的目标：产出 5-10 个候选方向**

### Phase 2: Convergent（收敛）

目标是缩小范围，选出最佳方案。

4. **建立评估标准**
   - 可行性（技术/时间/资源）
   - 影响程度（解决多少问题）
   - 风险等级
   - 与长期目标的一致性

5. **评分与排序** — 对每个候选方案按标准打分
   - 不需要复杂加权，定性分析即可
   - 标记每个方案的优缺点

6. **选择最佳方案** — 推荐 1-2 个方案
   - 说明选择理由
   - 记录被淘汰方案及其原因

### Output：方案提案

将结果整理为结构化提案文档：

```
## 背景与目标
## 探索过程
### 发散阶段（候选方案列表）
### 收敛阶段（评估与选择）
## 推荐方案
### 方案描述
### 实施路径
### 风险评估
## 备选方案
```

## Common Rationalizations

| Rationalization | Reality |
|---|---|
| "第一个想法通常是最好的" | 统计表明，排在第三第四的想法常常更具创新性。 |
| "考虑太多方案会浪费时间" | 选择一个错误方案重做的成本远高于前期探索。 |
| "这个方案明显不行，不用列出来" | 发散阶段不允许提前筛选，否则会丢失替代路径。 |
| "我们一直这么做的" | 习惯路径可能是最好的陷阱。刻意探索反向方案。 |
| "分析太多了，直觉就够了" | 直觉适用于熟悉的领域，新问题需要结构化思考。 |

## Red Flags

- 发散阶段产出少于 5 个候选方向
- 过早否定某个方案（"这个太复杂"、"这个不现实"）
- 收敛阶段的评估标准不明确
- 没有记录被淘汰的方案及其原因
- 提案缺少风险分析
- 整个过程只有 1 个方案被认真考虑（等于没有探索）

## Verification

- [ ] 发散发散阶段产出了 >= 5 个候选方向
- [ ] 收敛阶段使用了明确的评估标准
- [ ] 已记录被淘汰的方案和原因
- [ ] 推荐方案有清晰的实施路径
- [ ] 已完成风险评估
- [ ] 提案文档结构完整（背景、探索过程、推荐方案、备选方案）
- [ ] 如果涉及决策，用户已确认方向
