# 任务规划与分解

## Overview
将模糊的规格说明拆解为可独立实现、可验证的小任务。每个任务应能在一次改动中完成（~100 行），并有明确的完成标准。避免"大爆炸式"实现——先分解，再逐块攻克。

## When to Use
- 收到一份 PRD、需求文档或规格说明，需要落地为代码
- 面对一个大型功能，不知道该从哪里开始
- 需要给多个子 Agent 分配并行任务
- 评审他人计划时，检查任务粒度是否合理
- NOT for 已经小于 100 行改动的简单 bugfix
- NOT for 不需要拆解的纯配置变更

## Core Process

1. **理解全局** — 通读规格，标注不确定处，确认业务目标
2. **识别任务边界** — 按文件/模块/API 端点切分，每个任务聚焦单一职责
3. **确定依赖顺序** — 画 DAG：哪些任务必须先完成？哪些可以并行？
4. **编写验收标准** — 每个任务必须有可验证的完成条件（测试通过、UI 截图、API 响应）
5. **评估任务大小** — 每个任务代码改动控制在 ~100 行（±50）。过大则拆分，过小则合并
6. **标注优先级** — P0: 阻塞性基础、P1: 核心功能、P2: 增强/优化、P3: 低优先级
7. **评审计划** — 用怀疑的眼光检查：任务粒度是否均匀？依赖是否正确？验收标准是否可测？

## Common Rationalizations

| Rationalization | Reality |
|---|---|
| "这个需求很简单，直接写就行" | 简单不等于不需要分解。不分解的结果往往是写到一半发现遗漏 |
| "拆太细浪费时间" | 拆解花 10 分钟，节省 2 小时的返工 |
| "先写完再拆也一样" | 后拆分等于重构，成本远高于先拆 |
| "这个任务很难拆，它就是一个整体" | 没有不能拆的任务。拆不动说明理解不够 |

## Red Flags
- 某个任务的描述超过 3 句话
- 一个任务中出现了 "同时"、"以及"、"分别" 等词
- 开始写代码时才发现有个任务没考虑到
- 任务验收标准写的是 "代码能跑"
- 没有为 P0/P1 任务设置明确的前置依赖检查

## Verification
- [ ] 每个任务都有自己的验收标准（非空、可验证）
- [ ] 任务依赖 DAG 已画出（至少在脑内）
- [ ] 没有单个任务超过 200 行改动
- [ ] P0 任务清晰标记，且无遗漏
- [ ] 每个任务都有归属的文件/模块名
