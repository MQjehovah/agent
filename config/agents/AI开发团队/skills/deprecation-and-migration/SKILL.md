# 废弃与迁移

## Overview

代码是负债，不是资产。每行代码都是未来需要维护、理解、测试的负担。主动清理僵尸代码和废弃系统是保持代码库健康的关键。

## When to Use

- 系统需要下线或替换时
- 发现不再被调用的代码时
- API 版本需要退役时
- 用户需要从旧系统迁移到新系统时

- NOT for：重构（重构是重写，废弃是删除）
- NOT for：功能开发（不要一边加新功能一边堆积废弃代码）

## Core Process

1. **标记废弃（Mark Deprecated）** — 在代码/文档中明确标注 `@deprecated`，说明替代方案和预期移除时间
2. **沟通时间线（Communicate Timeline）** — 向所有利益相关者（用户、团队、依赖方）发出迁移通知，给予充分过渡期
3. **迁移用户（Migrate Users）** — 提供迁移指南和工具，帮助用户平滑过渡。监控迁移进度，识别并协助滞留用户
4. **移除代码（Remove）** — 按计划删除废弃代码。永远不保留注释掉的代码

**两种废弃模式：**
- **强制型（Compulsory）** — 给定日期后废弃功能不再可用，适用于内部 API 或可强制执行迁移的场景
- **建议型（Advisory）** — 功能仍可用但建议迁移，适用于外部用户或难以强制迁移的场景

## Common Rationalizations

| Rationalization | Reality |
|---|---|
| "留着也无害，万一有人用" | 代码增加编译时间、测试成本、认知负载、攻击面 |
| "先注释掉，以后可能要" | Git 历史就是你的"注释"，删掉 git 还能找回 |
| "迁移太麻烦，就让它一直跑着" | 旧系统总要维护，越晚迁移成本越高 |
| "用户说不想迁移" | 倾听但保持坚定，提供足够支持让迁移更容易 |

## Red Flags

- 代码库中有被 `#if 0` 或注释掉的大段代码
- 项目存在多个版本长期并存且无废弃时间表
- 搜索发现某个函数/接口没有任何调用方（死代码）
- 用户仍在依赖一个两年前就标记废弃的功能
- 迁移计划反复推迟无明确截止日期

## Verification

- [ ] 废弃代码已标记 `@deprecated` 并说明替代方案
- [ ] 迁移计划有时间线和截止日期
- [ ] 所有利益相关者已收到通知
- [ ] 废弃代码已确认无调用方后删除
- [ ] 避免了注释掉的代码留在代码库
- [ ] 迁移完成后关闭旧系统和相关基础设施
