# 安全与加固

## 概述
防止常见安全漏洞，保护用户数据和系统完整性。涵盖 OWASP Top 10、密钥管理、输入验证、依赖审计等安全实践。

## 何时使用
- 处理用户输入、认证、数据存储或外部集成时
- 编写涉及敏感数据或权限控制的代码
- 添加第三方依赖或调用外部 API
- NOT for 内部工具忽略安全——内部工具也会被攻破

## 核心流程
1. **OWASP Top 10 检查**：检查注入、失效认证、XSS、不安全直接对象引用、安全配置错误、敏感数据泄露、失效访问控制、CSRF、使用已知漏洞组件、未验证的重定向
2. **密钥管理**：永远不硬编码密钥——使用环境变量 `.env` 或密钥管理服务；使用 `.env.example` 作为模板，`.env` 加入 `.gitignore`
3. **输入验证**：在信任边界验证所有输入——白名单验证、参数化查询、输出编码
4. **依赖审计**：运行 `npm audit` 或 `pip audit` 并修复高危漏洞；定期更新依赖
5. **SQL 注入防护**：始终使用参数化查询或 ORM，绝不拼接 SQL 字符串
6. **安全头**：设置 CSP、X-Frame-Options、HSTS 等 HTTP 安全头

## 常见自我合理化
| 合理化 | 现实 |
|---|---|
| 这只是内部工具 | 内部工具也会被攻破——内部攻击是最常见的攻击向量之一 |
| 用户输入不会包含恶意内容 | 永远假设所有输入都是恶意的 |
| 这个密钥只有我看得见 | 代码迟早会被共享或泄露 |
| 先上线再修安全 | 安全不是事后补丁，是设计的一部分 |
| 这个依赖很流行不会有问题 | 流行度不等于安全性 |

## 红旗标志
- 密钥直接写在源代码中
- SQL 语句使用字符串拼接
- 不对用户输入做任何校验或转义
- 使用 `eval()` 或类似危险函数
- 禁用安全工具告警而不修复
- 说"这只是一个内部工具"

## 验证清单
- [ ] 密钥和密码使用环境变量，不在代码中硬编码
- [ ] 所有用户输入在信任边界验证和转义
- [ ] 使用参数化查询或 ORM 访问数据库
- [ ] 依赖审计无高危漏洞
- [ ] HTTP 安全头已配置
- [ ] 认证会话使用安全、HttpOnly Cookie
- [ ] 敏感数据在传输和存储时加密
- [ ] 禁用目录列表和服务器信息泄露
