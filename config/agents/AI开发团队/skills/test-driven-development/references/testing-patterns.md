# 测试模式快速参考

## 测试结构（AAA 模式）

```
Arrange  — 准备测试数据和前置条件
Act      — 执行被测操作
Assert   — 验证结果是否符合预期
```

## 命名规范

- 使用 `describe` / `it` 嵌套描述测试场景
- 测试名称应具描述性：`describe('when user is logged in')` / `it('should return 200 with profile data')`
- 名称要表达被测行为，而非实现细节

## Mock 使用指南

优先顺序（从最优先到最不优先）：

1. **真实实现** — 能用真实对象就用真实对象
2. **Fake** — 轻量级替代实现（如内存数据库）
3. **Stub** — 返回固定值的替身
4. **Mock** — 验证交互行为的替身

## DAMP > DRY

- 测试代码优先保证 **可读性** 和 **可理解性**
- 允许适当的重复（DAMP），不要过度抽取公用方法
- 每个测试应能独立阅读，无需跳转上下文

## React Testing Library 模式

- 优先通过用户可见的文本查询元素（`getByRole`、`getByText`、`getByLabelText`）
- 避免测试实现细节（state、内部方法）
- 使用 `waitFor` / `findBy` 处理异步更新
- 优先 `userEvent` 而非 `fireEvent`

## API 测试模式

- 验证 HTTP 状态码和响应体结构
- 测试正常路径和错误路径
- 使用工厂函数（Factory）构建测试数据
- 每个测试独立，不依赖其他测试的副作用

## 反模式清单

- ❌ 测试依赖执行顺序
- ❌ 测试访问私有方法/内部状态
- ❌ 过度 Mock 导致测试与实现紧耦合
- ❌ 测试过于复杂，难以理解意图
- ❌ 忽略边界条件和异常分支
- ❌ 在断言中使用魔法数字
