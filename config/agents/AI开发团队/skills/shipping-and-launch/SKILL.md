# 发布与上线

## Overview

发布是高风险操作。标准化的预发布检查清单和分阶段上线流程，确保每次发布都可控、可回滚、可观察。

## When to Use

- 新功能准备部署到生产环境
- 重大重构或基础架构变更上线
- 每周常规发布
- 紧急修复（hotfix）

- NOT for：开发环境的日常部署
- NOT for：仅涉及 UI 文案修改的发布（但仍需 B 轮验证）

## Core Process

1. **预发布检查清单** — 确认以下事项全部通过：
   - 所有测试通过（单元测试、集成测试、E2E 测试）
   - 安全审计无高危漏洞
   - 性能基准测试无退化
   - 回滚方案就绪（数据库迁移可回退）
   - 监控仪表板和告警规则已配置
   - 发布通知已发送给相关方
2. **Feature Flag 生命周期** — 新功能默认关闭 → 开发环境启用 → 内部测试 → 灰度发布 → 全量开放 → 移除 Flag 代码
3. **分阶段发布（Staged Rollout）** — 1% → 5% → 25% → 50% → 100%。每个阶段观察 5-15 分钟，确认 RED 指标正常后进入下一阶段
4. **回滚流程** — 定义明确的回滚触发条件和执行步骤。回滚应一键执行，无需思考。数据库变更必须向前兼容

## Common Rationalizations

| Rationalization | Reality |
|---|---|
| "就改了一行代码，不用走流程" | 90% 的线上事故来自"就改了一行" |
| "功能 Flag 太麻烦，直接发" | 没有 Flag 就无法灰度，出问题只能全量回滚 |
| "1% 测不出问题" | 1% 能发现配置类和大规模影响的问题 |
| "这次不用回滚方案，不会有问题" | 没有回滚计划的发布就不是发布，是赌博 |

## Red Flags

- 发布前没有回滚方案
- 数据库变更不兼容旧版本代码（无法回滚）
- 发布日当天下午才决定上线
- 功能 Flag 在全量开放后未被移除
- 告警规则在发布后才配置
- 发布后无人值守监控

## Verification

- [ ] 预发布检查清单全部项已确认通过
- [ ] 回滚方案已定义并通过演练
- [ ] Feature Flag 已配置，初始状态为关闭
- [ ] 分阶段发布计划已设定（1% → 5% → 25% → 50% → 100%）
- [ ] 监控仪表板和告警规则已就绪
- [ ] 数据库变更向前兼容，可安全回滚
- [ ] 权益相关方已收到发布通知
